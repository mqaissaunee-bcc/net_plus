Windows provides powerful built-in networking tools through the Command Prompt and PowerShell. These commands are essential for Windows-based network administration and troubleshooting.

Command Purpose Key Parameters Example Usage
ipconfig Display/manage IP configuration /all, /release, /renew, /flushdns ipconfig /all
ping Test connectivity to remote host -t, -n count, -l size, -i TTL ping -t google.com

üîÑ Command Chaining and Scripting

Combining commands creates powerful troubleshooting sequences. Here are practical examples:

Windows Command Sequences

REM Complete network diagnostics sequence ipconfig /all > network_info.txt ping -n 4 8.8.8.8 >> network_info.txt tracert 8.8.8.8 >> network_info.txt nslookup google.com >> network_info.txt netstat -rn >> network_info.txt echo Network diagnostics complete! REM Reset network configuration ipconfig /release ipconfig /flushdns ipconfig /renew netsh winsock reset netsh int ip reset

Linux Command Sequences

#!/bin/bash # Comprehensive network check script echo "=== Network Interface Information ===" ip addr show echo -e "\n=== Routing Table ===" ip route show echo -e "\n=== DNS Configuration ===" cat /etc/resolv.conf echo -e "\n=== Connectivity Test ===" ping -c 4 8.8.8.8 echo -e "\n=== DNS Resolution Test ===" dig google.com +short echo -e "\n=== Active Connections ===" ss -tuln

üìä Output Interpretation Guide

Understanding ping Output

Key Metrics:
‚Ä¢ Reply time: Network latency (lower is better)
‚Ä¢ TTL (Time To Live): Remaining hops (helps identify OS)
‚Ä¢ Packet loss: Network reliability indicator
‚Ä¢ Request timeout: Indicates connectivity issues

Analyzing traceroute Results

Troubleshooting Tips:
‚Ä¢ * * * (asterisks): Router not responding to ICMP
‚Ä¢ High latency spike: Congestion at that hop
‚Ä¢ Consistent timeouts: Firewall or routing issue
‚Ä¢ Loops or long paths: Routing table problems

Reading Routing Tables

Routing table entries show how packets are forwarded:

  • Destination: Target network or host
  • Gateway: Next hop router
  • Interface: Outgoing network interface
  • Metric: Cost of the route
  • Route Source: How the route was learned

üéØ Common Troubleshooting Scenarios

Scenario 1: Cannot Access Internet

  1. Check local configuration: ipconfig /all (Windows) or ip addr show (Linux)
  2. Test local connectivity: ping 127.0.0.1 (loopback)
  3. Test gateway: ping [gateway_ip]
  4. Test external connectivity: ping 8.8.8.8
  5. Test DNS resolution: nslookup google.com
  6. Trace the route: tracert google.com

Scenario 2: Slow Network Performance

  1. Check interface statistics: netstat -i (Linux) or netsh int ip show int (Windows)
  2. Monitor active connections: netstat -an
  3. Test latency: ping -t destination
  4. Check for packet loss: Extended ping tests
  5. Analyze traffic: tcpdump (Linux) or Wireshark

üèÜ Best Practices

  • Document findings: Always save command output for analysis
  • Use systematic approach: Start with basic connectivity, then move to specific issues
  • Understand your baseline: Know normal network performance metrics
  • Test incrementally: Test each network layer systematically
  • Verify changes: Always confirm that fixes actually resolve the issue
  • Keep learning: Networks evolve, so keep updating your command knowledge
